# Prysm Design System

A small, composable set of primitives for **Prysm Health** — software for skilled
nursing facilities. The components here are generic (no patient/domain knowledge);
they carry the product's visual language — Figtree type, a slate neutral spine with
a single teal action color, semantic `tone`s, lucide icons, and **redundant
encoding** (state is never color alone).

This page and **[Choosing a component](?path=/docs/choosing-a-component--docs)**
exist so you (and tools reading these docs) can pick the right primitive without
guessing. Every component also carries a **"when to use / when not to use"** blurb
in its own Docs page.

---

## Principles that drive component choice

- **Redundant encoding, always.** No state is carried by color alone — color is
  paired with an icon and a text label (see **Badge**). It's a WCAG requirement and
  a reliability one.
- **Weight carries hierarchy, not color.** Importance comes from type weight/size;
  color is reserved for status and action. This is why **Field** leans on size, not
  tint, for emphasis.
- **Touch-friendly.** Interactive targets are ≥ **48px**. **Button**'s `size="touch"`
  and the **Stepper**/**Toggle** controls are sized for tablets and gloved hands.
- **Hairlines over shadows.** Separation is done with 1px borders; elevation is spent
  only on things that genuinely float (**OverlayPanel**).

---

## Prefer structured input

Where the data is structured — numeric or a predefined selection — reach for the
structured control rather than a freeform field. The product captures clinical data
(vitals, ADLs, meal intake) that must stay clean, comparable, and fast to enter
mid-task, so a constrained control beats a text box for those values:

| You need to capture… | Use | Prefer over |
|---|---|---|
| A number (vital, count, 0–10 scale) | **Stepper** | **TextInput** (freeform) |
| Exactly one of a small fixed set | **Segmented** | **Select** (better for longer lists) |
| A binary on/off state | **Toggle** | a checkbox-as-text |

Freeform inputs do exist — **TextInput**, **TextArea**, and **Select** — for genuinely
unstructured or long-list values; reach for them only when a structured control
doesn't fit. Wrap any control in **FormField** to give it a label (and an optional
"last value" reference).

**Remaining deliberate gap:** there is no dedicated **multi-select** control — for
several independent on/off states, compose multiple **Toggle**s (one per option).

## The catalog at a glance

- **Actions** — **Button** (the only interactive action primitive).
- **Forms** — **Segmented** (one-of-N), **Toggle** (binary), **Stepper** (numeric),
  **Select** (one-of-many), **TextInput** / **TextArea** (freeform text),
  **FormField** (labels a control + optional reference value).
- **Data Display** — **Badge**, **Card**, **Section**, **Field**, **IconField**,
  **IconTile**, **Avatar**, **TaskProgress**, **ColumnHeader**, **SortIndicator**,
  **EntityCard** / **EntityRow** (a record at card- and list-density).
- **Feedback** — **Skeleton** (loading), **EmptyState** (no results),
  **SyncStatus** (save / sync state).
- **Overlays** — **OverlayPanel** (covering detail/drawer).
- **Navigation** — **AppBar** (minimal top-bar shell).
- **Keyboard** — **Kbd** (shortcut hints).
- **Tokens** — Color Tones.

Confusable pairs (Badge vs IconTile vs Button-as-chip, Field vs IconField,
Card vs Section, EntityCard vs EntityRow, TextInput vs Stepper vs Select,
Segmented vs multi-select) are disambiguated in
**[Choosing a component](?path=/docs/choosing-a-component--docs)**.
